架构方案对比

三种实现方案，从简到繁

A

轻量 MVVM + JSON 文件存储

NavigationSplitView 布局，ObservableObject ViewModel，聊天数据存本地 JSON 文件。最快落地，但扩展性有限。

Pros

  • 实现最快，文件少
  • 无外部依赖

Cons

  • 数据量大时性能差
  • 后续接远端需大改
B

MVVM + SwiftData 持久化 ✅ 推荐

使用 Apple SwiftData 框架管理数据模型，@Observable 宏实现响应式。内置 iCloud 同步潜力，适合长期维护。

Pros

  • Apple 原生方案，与 SwiftUI 深度集成
  • 自动数据迁移和查询优化
  • 未来可无缝开启 iCloud 同步

Cons

  • 需要 iOS 17+ / macOS 14+
  • SwiftData 仍在快速迭代
C

Clean Architecture + 协议抽象存储层

分层架构：View → ViewModel → UseCase → Repository（协议）。本地存储可随时切换为远端 API。

Pros

  • 最佳扩展性，换存储不影响上层
  • 可测试性最强

Cons

  • 初期代码量大
  • 当前阶段可能过度设计
