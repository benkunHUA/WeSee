# 智能体对话记录 PostgreSQL 持久化设计

日期：2026-06-25

## 目标

将智能体对话记录持久化到 PostgreSQL，使服务端在进程重启后仍能回放聊天历史并继续对话。本设计第一阶段只覆盖核心聊天历史：会话、用户消息、最终 assistant 回复，以及轻量工具调用摘要。本阶段不保存完整 token 流、thinking 事件或完整智能体运行轨迹。

## 范围

本阶段包含：

- 使用 PostgreSQL 作为服务端聊天历史的权威数据源。
- 引入 Alembic 管理数据库 schema 迁移。
- 在应用尚无登录系统时，使用稳定的默认本地用户。
- 触发 `new_conversation` 时创建新会话并保留旧对话。
- HTTP API 与 WebSocket 重连/恢复流程从 PostgreSQL 加载消息历史。
- streaming 正常完成后，持久化最终 assistant 内容。

本阶段不包含：

- 完整智能体 run/event 追踪。
- 保存每个流式 token、thinking 分片或中间服务端事件。
- 真实多用户认证与授权。
- 与 macOS SwiftData 本地缓存做复杂双向同步。
- 搜索、保留策略或分析看板。

## 设计原则

- PostgreSQL 是权威数据源；内存 session 只作为热缓存。
- 对话历史在消息层面采用追加写入。
- 新对话创建新 session，不删除旧历史。
- 数据库写入采用同步 write-through，优先保证可靠性。
- 持久化逻辑通过 repository/service 边界隔离，handler 不直接嵌入 SQLAlchemy 细节。
- schema 选择优先保证 PostgreSQL 正确性，同时贴近现有 Python 模型边界。

## 数据模型

### `users`

当前应用没有登录系统。服务端会创建或复用一个稳定的默认用户：

- `id text primary key`，初始值为 `local-user`。
- `name text not null`。
- `created_at timestamptz not null`。

这样既不增加认证复杂度，也为未来接入真实用户系统保留清晰边界。

### `sessions`

一条 session 表示一次可恢复的持久化对话。

字段：

- `id text primary key`：UUID 字符串，匹配当前运行时 session ID，也便于测试。
- `user_id text not null references users(id)`。
- `title text null`：可选会话标题。第一版可以留空，也可以由首条用户消息截断生成。
- `workspace_path text not null`。
- `status text not null`：初始支持 `active` / `archived`。
- `created_at timestamptz not null`。
- `updated_at timestamptz not null`。
- `last_message_at timestamptz null`。

索引：

- `(user_id, updated_at desc)`：用于会话列表。
- `(status, updated_at desc)`：用于 active/archive 筛选。

`new_conversation` 会创建新的 `sessions` 记录并返回新的 `session_id`，不会删除或清空上一个 session 的消息。

### `messages`

一条 message 表示一条可回放的聊天上下文消息。

字段：

- `id bigint generated by default as identity primary key`。
- `session_id text not null references sessions(id)`。
- `sequence bigint not null`：同一个 session 内单调递增。
- `role text not null`：限制为 `user`、`assistant`、`system` 或 `tool`。
- `content text not null`。
- `tool_calls jsonb null`：保存 assistant 工具调用的轻量可回放摘要。
- `tool_call_id text null`。
- `name text null`。
- `metadata jsonb not null default '{}'::jsonb`：保存少量扩展信息，例如模型名、客户端来源或附件摘要。
- `created_at timestamptz not null`。

约束与索引：

- `unique(session_id, sequence)`。
- `(session_id, sequence)`：用于稳定加载历史。
- `(session_id, id)`：兼容当前按时间顺序读取的思路。

只有在出现明确查询需求后，才为 `metadata` 或 `tool_calls` 增加 GIN 索引，避免过早增加写入和维护成本。

## 运行时架构

### Conversation repository/service

新增持久化边界，命名为 `ConversationStore` 或类似名称。它提供以下能力：

- `ensure_default_user()`。
- `create_session(user_id, workspace_path)`。
- `get_session(session_id)`。
- `list_sessions(user_id)`。
- `append_message(session_id, message)`。
- `load_messages(session_id)`。
- `touch_session(session_id, last_message_at)`。

HTTP 和 WebSocket handler 只调用这个边界，不直接操作 SQLAlchemy session。

### Session manager

当前内存 `SessionManager` 仍作为热缓存保留，但不再作为权威数据源。它需要扩展或被包装，以支持：

- 为已持久化的 `sessions` 记录创建运行时 session。
- resume 时把 PostgreSQL 中的消息加载到运行时 `Session`。
- WebSocket 断开时只释放内存缓存，不删除 PostgreSQL 记录。

### 应用生命周期

FastAPI 启动时根据 `POSTGRES_DSN` 初始化 async SQLAlchemy engine 和 session factory。schema 变更由 Alembic migration 管理。如果 PostgreSQL 不可用或迁移状态不正确，启动应明确失败；不能静默退回到纯内存模式，以免用户误以为历史已经持久化。

## 数据流

### HTTP chat

1. 解析 `local-user` 当前 active session；没有则创建一个。
2. 同步持久化用户消息。
3. 将用户消息加入内存缓存。
4. 使用该 session 的历史运行 agent。
5. 聚合 streaming assistant token，形成最终 assistant 内容。
6. 收到 `done` 后，同步持久化最终 assistant 消息。
7. 更新 `sessions.updated_at` 和 `sessions.last_message_at`。
8. `/api/messages` 从 PostgreSQL 读取，而不是只枚举内存消息。

### HTTP new conversation

1. 为 `local-user` 创建新的 `sessions` 记录。
2. 将新的 session 设为当前 HTTP session。
3. 返回新的 `session_id`。
4. 保留旧 session 和旧消息。

### WebSocket connect 与 resume

连接建立时：

- 如果客户端提供 `session_id`，从 PostgreSQL 加载该 session 及其消息。
- 如果客户端没有提供 `session_id`，创建新 session。
- 将当前 `session_id` 发回客户端，供后续重连使用。

聊天时：

1. 持久化用户消息。
2. 使用已加载的 session 历史运行 agent。
3. 向客户端流式发送事件，同时在服务端聚合 assistant 内容。
4. 收到 `done` 后，持久化最终 assistant 消息。
5. 连接断开时，只移除内存缓存。

### macOS SwiftData 缓存

PostgreSQL 是权威数据源。SwiftData 继续作为本地 UI/offline 缓存。本阶段不实现冲突解决或完整双向同步。恢复会话时，应以服务端加载的历史作为 canonical history。

## 一致性与并发

写入采用同步 write-through：

- 用户消息必须先成功入库，才调用 agent。
- assistant 消息只在 agent 成功完成后入库。
- 如果 agent 失败，保留用户消息，但不写入虚假的 assistant 回复。

消息排序使用 `sequence`，不依赖时间戳排序。`append_message` 应在事务内分配下一个 sequence。为处理同一 session 的并发写入，repository 应锁定 session 行，或在 `unique(session_id, sequence)` 冲突时重试。

## 错误处理

- 数据库启动失败：启动失败，并明确提示 `POSTGRES_DSN` 和迁移状态。
- 用户消息写入失败：返回明确的 API/WebSocket 错误，不调用 agent。
- agent 成功但 assistant 消息写入失败：报告并记录持久化失败，避免客户端误以为回复已持久化。
- Alembic migration 失败：启动或测试 setup 立即失败。
- 无效 session ID：返回清晰的 not-found/invalid-session 错误，不静默创建无关的新 session。

## 测试策略

实现阶段遵循 TDD。

Repository 测试：

- 创建默认用户和 session。
- append/load messages 时保持 `sequence` 顺序。
- 创建新对话时保留旧 session。
- 保持 timezone-aware UTC datetime。
- 验证 role 约束与 session 外键。

PostgreSQL 集成测试：

- 使用 testcontainers 在真实 PostgreSQL 上运行 Alembic migrations。
- 验证 JSONB 字段、唯一约束、外键和按顺序加载历史。
- 验证新 app/session manager 实例可以重新加载已有会话。

HTTP 测试：

- `/api/chat` 持久化用户消息和最终 assistant 消息。
- `/api/messages` 从 PostgreSQL 读取。
- `/api/new-conversation` 返回新的 `session_id`，并保留旧历史。

WebSocket 测试：

- 首次连接获得 `session_id`。
- 使用 `session_id` 重连可以恢复历史。
- streaming `done` 后持久化最终 assistant 内容。
- 断开连接只释放内存，不删除数据库记录。

客户端回归测试：

- 新的 session-id 协议不破坏现有 Swift 聊天基础流程。
- resume 会话时以服务端历史为 canonical history。

## 迁移策略

在 server 包中引入 Alembic。初始 migration 创建 `users`、`sessions` 和 `messages`，包含本设计定义的字段、约束和索引。现有 SQLAlchemy model 需要同步更新，使其与 migration 一致，包括在合适位置使用 PostgreSQL `jsonb`。

因为当前项目只有内存态服务端历史，所以没有既有服务端对话数据需要回填。现有 SwiftData 消息仍保留为客户端本地缓存，本阶段不迁移。

## 后续扩展

本 schema 有意把完整运行轨迹排除在第一阶段之外。后续可以增加：

- `agent_runs`：保存模型、token 用量、耗时、状态和错误。
- `conversation_events`：保存 token/thinking/tool event 以支持事件级 replay。
- 消息内容全文搜索。
- 真实用户与授权检查。
- 保留/归档策略。
- 更好的标题生成和会话列表 API。
