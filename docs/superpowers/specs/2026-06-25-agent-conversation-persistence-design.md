# Agent Conversation PostgreSQL Persistence Design

Date: 2026-06-25

## Goal

Persist agent conversation records to PostgreSQL so the server can replay chat history and continue conversations after process restarts. This design intentionally targets the core chat-history layer first: sessions, user messages, final assistant messages, and lightweight tool-call summaries. It does not store full token streams, thinking events, or complete agent run traces in this phase.

## Scope

In scope:

- Use PostgreSQL as the server-side source of truth for chat history.
- Introduce Alembic-managed schema migrations.
- Keep a stable default local user while the app has no authentication system.
- Preserve old conversations when `new_conversation` is triggered by creating a new session.
- Load message history from PostgreSQL for HTTP APIs and WebSocket reconnect/resume flows.
- Persist final assistant content after streaming completes successfully.

Out of scope for this phase:

- Full agent run/event tracing.
- Persisting every streamed token, thinking chunk, or intermediate server event.
- Real multi-user authentication and authorization.
- Complex bidirectional synchronization with the macOS SwiftData cache.
- Search, retention policies, or analytics dashboards.

## Design Principles

- PostgreSQL is authoritative; in-memory sessions are hot caches only.
- Conversation history is append-only at the message level.
- A new conversation creates a new session instead of deleting prior history.
- Database writes are synchronous write-through for reliability.
- The persistence layer is isolated behind a repository/service API so handlers do not embed SQLAlchemy details.
- Schema choices favor PostgreSQL correctness while remaining close to the existing Python model boundaries.

## Data Model

### `users`

The application currently has no login system. The server will create or reuse one stable default user:

- `id text primary key`, initially `local-user`.
- `name text not null`.
- `created_at timestamptz not null`.

This keeps the schema ready for future real users without adding authentication complexity now.

### `sessions`

A session is one durable conversation that can be resumed.

Columns:

- `id text primary key` — UUID string, matching the current runtime session IDs and keeping tests simple.
- `user_id text not null references users(id)`.
- `title text null` — optional conversation title. The first implementation may leave this empty or derive it from the first user message.
- `workspace_path text not null`.
- `status text not null` — initially `active` or `archived`.
- `created_at timestamptz not null`.
- `updated_at timestamptz not null`.
- `last_message_at timestamptz null`.

Indexes:

- `(user_id, updated_at desc)` for conversation lists.
- `(status, updated_at desc)` for active/archive filtering.

`new_conversation` creates a new `sessions` row and returns its `session_id`. It does not delete or clear messages from the previous session.

### `messages`

A message is one replayable chat-context item.

Columns:

- `id bigint generated by default as identity primary key`.
- `session_id text not null references sessions(id)`.
- `sequence bigint not null` — monotonically increasing within each session.
- `role text not null` — constrained to `user`, `assistant`, `system`, or `tool`.
- `content text not null`.
- `tool_calls jsonb null` — lightweight assistant tool-call summaries when needed for replay.
- `tool_call_id text null`.
- `name text null`.
- `metadata jsonb not null default '{}'::jsonb` — small extension data such as model name, client source, or attachment summaries.
- `created_at timestamptz not null`.

Constraints and indexes:

- `unique(session_id, sequence)`.
- `(session_id, sequence)` for stable history loading.
- `(session_id, id)` to support current chronological lookup patterns.

GIN indexes on `metadata` or `tool_calls` are deferred until there is a concrete query need.

## Runtime Architecture

### Conversation repository/service

Add a persistence boundary, named `ConversationStore` or equivalent, with methods such as:

- `ensure_default_user()`.
- `create_session(user_id, workspace_path)`.
- `get_session(session_id)`.
- `list_sessions(user_id)`.
- `append_message(session_id, message)`.
- `load_messages(session_id)`.
- `touch_session(session_id, last_message_at)`.

Handlers should call this boundary instead of directly using SQLAlchemy sessions.

### Session manager

The current in-memory `SessionManager` remains useful as a hot cache, but it is no longer the source of truth. It should be extended or wrapped so it can:

- Create a runtime session for a persisted `sessions` row.
- Load persisted messages into the runtime `Session` when resuming.
- Release memory on WebSocket disconnect without deleting PostgreSQL records.

### Application lifecycle

FastAPI startup should initialize the async SQLAlchemy engine and session factory from `POSTGRES_DSN`. Schema changes should be managed through Alembic migrations. Startup must fail clearly if PostgreSQL is unavailable or migrations are not in the expected state; it should not silently fall back to memory-only persistence.

## Data Flow

### HTTP chat

1. Resolve the current active session for `local-user`; create one if needed.
2. Persist the user message synchronously.
3. Add the user message to the in-memory cache.
4. Run the agent with the session history.
5. Aggregate streamed assistant tokens into final assistant content.
6. On `done`, persist the final assistant message synchronously.
7. Update `sessions.updated_at` and `sessions.last_message_at`.
8. `/api/messages` reads from PostgreSQL instead of enumerating only in-memory messages.

### HTTP new conversation

1. Create a new `sessions` row for `local-user`.
2. Make that new session the current HTTP session.
3. Return the new `session_id`.
4. Leave previous sessions and messages intact.

### WebSocket connect and resume

On connect:

- If the client provides a `session_id`, load that session and its messages from PostgreSQL.
- If no `session_id` is provided, create a new session.
- Send the active `session_id` back to the client so it can reconnect later.

On chat:

1. Persist the user message.
2. Run the agent with the loaded session history.
3. Stream events to the client while aggregating assistant content server-side.
4. On `done`, persist the final assistant message.
5. On disconnect, remove only the in-memory cache entry.

### macOS SwiftData cache

PostgreSQL is authoritative. SwiftData remains a local UI/offline cache. This phase does not implement conflict resolution or full two-way sync. Server-loaded history should be treated as the canonical history when resuming a conversation.

## Consistency and Concurrency

Writes use synchronous write-through semantics:

- A user message must be stored before the agent is invoked.
- An assistant message is stored only after the agent finishes successfully.
- If the agent fails, the user message remains, but no fake assistant response is stored.

Message ordering uses `sequence`, not timestamp ordering. `append_message` should allocate the next sequence inside a transaction. To handle concurrent writes to the same session, the repository should lock the session row or retry on `unique(session_id, sequence)` conflicts.

## Error Handling

- Database startup failure: stop startup with a clear error mentioning `POSTGRES_DSN` and migration state.
- User-message write failure: return an explicit API/WebSocket error and do not invoke the agent.
- Assistant-message write failure after a successful agent run: report/log that persistence failed so the client does not assume the response is durable.
- Alembic migration failure: fail startup or the test setup immediately.
- Invalid session ID: return a clear not-found/invalid-session error instead of creating an unrelated session silently.

## Testing Strategy

Follow TDD for implementation.

Repository tests:

- Create default user and session.
- Append and load messages in `sequence` order.
- Preserve old sessions when creating a new conversation.
- Preserve timezone-aware UTC datetimes.
- Validate role constraints and session foreign keys.

PostgreSQL integration tests:

- Run Alembic migrations against real PostgreSQL via testcontainers.
- Verify JSONB columns, unique constraints, foreign keys, and chronological loading.
- Verify reconnect/reload behavior across a fresh app/session manager instance.

HTTP tests:

- `/api/chat` persists the user message and final assistant message.
- `/api/messages` reads from PostgreSQL.
- `/api/new-conversation` returns a new `session_id` and preserves old history.

WebSocket tests:

- First connection receives a `session_id`.
- Reconnect with `session_id` reloads persisted history.
- Streaming `done` persists final assistant content.
- Disconnect releases memory without deleting database records.

Client regression tests:

- Existing Swift chat flow still works with the new session-id protocol.
- Server history is treated as canonical when a session is resumed.

## Migration Strategy

Introduce Alembic under the server package. The initial migration should create `users`, `sessions`, and `messages` with the columns, constraints, and indexes described above. Existing SQLAlchemy model definitions should be updated to match the migration, including PostgreSQL `jsonb` usage where appropriate.

Because this project currently has only in-memory server history, there is no existing server-side conversation data to backfill. Existing SwiftData messages remain client-local cache data and are not migrated in this phase.

## Future Extensions

The schema deliberately leaves full run tracing out of phase one. Future phases can add:

- `agent_runs` for model, token usage, duration, status, and errors.
- `conversation_events` for token/thinking/tool event replay.
- Full-text search over message content.
- Real users and authorization checks.
- Retention/archive policies.
- Better title generation and session list APIs.
